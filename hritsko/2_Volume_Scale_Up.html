Architecture Foundation
The Unchanging Core

The system has one spine and three shells. The spine never gets replaced. Every upgrade is additive — a layer on top, never a rebuild underneath. Design these five components right once. They survive 10, 60, and 120 orders per day identically.

Design test. If adding a second operator requires a new system — not a new column — the core was built wrong. Everything below passes this test.
🏷
Batch ID system
Every unit produced gets a batch ID. Format: SKU-YYMMDD-###. Written on label, logged in sheet. One format forever.
Never changes: the format, the logic, and the rule that nothing ships without a batch ID present.
📋
Packaging checklist
10 steps. Every order, every time. Verify SKU → check qty → pack → apply label → confirm batch ID → QA gate → seal → log → stage. The checklist is the process, not the person's memory.
Never changes: the steps. What changes is who owns each step and whether a scan or a person does the verifying.
🔍
3-step QA checkpoint
Pre-pack (right product?), mid-pack (right quantity?), pre-ship (right label + batch ID?). Three gates. Nothing leaves without clearing all three. No exceptions.
Never changes: the three gates. What changes is whether a person or a barcode scan does the checking.
📊
Order → batch → shipment log
Every order links to a batch ID. Every batch ID links to a shipment record. One row equals one order. This is both the audit trail and the daily ops dashboard simultaneously.
Never changes: the linkage model. What changes is whether it lives in a Sheet or a dedicated tool.
👤
Named ownership per order
One person is accountable for every order reaching shipped status. At low volume it's Leighanna. At high volume it's a team lead. Accountability never diffuses into "the team."
Never changes: clear ownership per order. What changes is how many named owners exist in the system.
What Never Changes — Across All Volumes
Stable Rules
SKU-YYMMDD-### batch ID format 10-step packaging checklist 3-gate QA logic (pre / mid / pre-ship) Order → batch → shipment linkage One named owner per order Nothing ships without a batch ID Daily close-out log review Leighanna owns operations outcome
The test of a well-designed core. At 10 orders/day or 120 orders/day, a new operator handed the checklist and the sheet can execute every step correctly without asking questions. That is the target.

Risk Architecture
The Three Break Points

Every system has predictable failure modes. These three are structural — they occur at specific thresholds regardless of how well the team executes. Design around them before hitting them, not after.

1
Single-operator ceiling
Hits at ~25 orders/day
One person doing packing + QA + logging + managing exceptions + communicating with Eric is a 4-hat job. It works at 15 orders/day. It collapses at 25. The fix is not working faster — it is adding an operator and separating roles. This is not a performance problem; it is a structural limit.
Symptom
Pack time consistently over 3h/day. Leighanna is still thinking about open orders after hours.
Fix
Hire packaging associate. Leighanna moves to QA + oversight immediately — not gradually.
2
Google Sheet coordination ceiling
Hits at 2 operators + ~50/day
Two people simultaneously updating one Google Sheet creates race conditions — overwritten rows, missed status changes, duplicate batch IDs. The Sheet was designed for one brain. At two operators it needs rules. At three operators it needs a dedicated routing tool. This is the Sheet's designed limit, not a flaw.
Symptom
Sheet errors appearing 3+ times per week. Operators asking "did you log that?" regularly.
Fix
Add ShipStation as order router. Sheet becomes verification layer — not the primary entry point.
3
Manual QA ceiling
Hits at 3 operators + ~80/day
Visual QA works when one person can hold the entire order set in their head. At 80+ orders/day across 3 operators, no human can audit everything visually. One operator skipping a QA gate becomes undetectable without a scan record. This is where a mislabeled shipment reaches a customer — not because someone made a mistake, but because the system had no enforcement mechanism.
Symptom
First customer complaint about wrong product or missing item. Or Leighanna cannot confirm whether a specific order was QA-checked.
Fix
Barcode scan at the pre-ship gate. Scan creates a timestamped record. QA becomes auditable — not trust-based.
The rule on upgrade triggers. These are thresholds, not suggestions. When the symptom appears — not before, not long after — execute the fix. Early upgrades add cost and complexity before the need exists. Late upgrades cause customer-facing failures. Hit the trigger, then move.
