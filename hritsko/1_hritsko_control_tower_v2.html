Program Overview
Operations Control Tower — v2

Hritsko is a scaling product business with a real bottleneck: packaging and fulfillment throughput. The founder is overloaded. Operations lack structure. This system exists to fix the bottleneck, create visibility, install repeatable control, and free Eric to sell.

What it solves. Packaging chaos, order visibility gaps, and reliance on memory instead of systems.
What it enables. Leighanna operating independently, Eric focused on sales, first hire onboarding cleanly.
How it scales. Core system unchanged at 2x–5x. Only layers around it evolve. Nothing gets rebuilt from scratch.

Decision Architecture
Assumption Layer
Operating principle. We do not block progress waiting for perfect information. Every assumption is labeled, reasonable, and reversible. If an assumption is wrong, the fix is a column or a tab — not a new system.
Order volume + velocity
Proceed
Assumption: 20–60 orders/week today, targeting 2x in 6 months. Reasonable for a small product business under scale pressure with a founder still in the fulfillment loop.
What we build on this
Single-operator Google Sheet. Manual batch log. Simple 3-step QA checkpoint.
If wrong — much higher volume
Sheet gets a 2nd operator tab. Batch logic expands. Hiring urgency increases immediately.
Tools in use today
Proceed
Assumption: Shopify for orders, no dedicated WMS, no ShipStation, order tracking in email or memory. This matches the "unstructured ops" description and is most common at this growth stage.
What we build on this
Google Sheets as the ops layer. Manual order pull from Shopify. CSV export as the bridge.
If wrong — has ShipStation
Sheet structure stays. We add a ShipStation sync column. Minimal rework — under 30 min.
SKU complexity + lot tracking
Proceed
Assumption: 5–20 active SKUs. Some have lot/expiration relevance (consumables). No current regulatory obligation for formal traceability, but risk exists. We design for it — cost is near zero.
What we build on this
Batch ID includes SKU + date + sequence. Lot field on every label. One row per batch in the log.
If wrong — more SKUs or complex
Batch ID logic extends. Label template gets a variant field. Sheet adds a SKU-lookup tab.
Current packaging team
Watch
Assumption: Leighanna + Eric are the current packaging team. No dedicated hourly worker yet. This is why throughput is bottlenecked — two people wearing too many hats.
What we build on this
Checklists designed for 1 operator. First hire role is packaging-first. Hiring chat starts this week.
If wrong — has hourly help
Checklists become training tools immediately. Hiring chat pivots to management, not recruiting.
Leighanna's biggest pain point
Proceed
Assumption: No visibility into order status and packaging inconsistency. She doesn't know what's been packed, what's pending, or what's shipped without checking manually each time.
What we build on this
Order status column in Google Sheet. Pack log with timestamp. Simple daily status view.
If wrong — different pain point
We pivot the sheet's primary view. Core structure unchanged — just reorder the columns.

Decision Engine
Priority Order — What to Decide When
Three decisions must be made today. They unlock 6+ downstream outputs. Waiting costs days of build time and compounds.
Decide today — unlocks 6 outputs
Lock the batch ID format
Everything downstream — labels, sheet, QA gates, traceability — is built around this. Recommended: SKU-YYMMDD-###. One decision unlocks the entire build. Override only if a specific format already exists.
Decide today — sets the ops layer
Google Sheet as v1 ops layer — confirm or override
Sheets is the assumption. If a tool already exists, we pivot now — not after building the Sheet structure. Five-minute decision that saves hours of rework.
Decide today — changes build target
Is Leighanna starting execution this week — yes or no
If yes, build everything for immediate handoff and hands-on use. If no, Phil builds the skeleton and Leighanna corrects remotely. Changes the format and urgency of every deliverable.
Decide this week
First hire: post now or wait 30 days
If volume is growing, waiting costs throughput. The role is defined regardless of volume. Posting takes 20 minutes. This decision affects week 2 planning entirely.
Decide this week
Label format: printed, handwritten, or hybrid
Drives the branding artifacts and QA checkpoint design. Low-cost decision but must be made before the label template is finalized.
Decide later — not urgent
Barcode / QR scan infrastructure
Right system at the wrong time. Manual first. Scan-based only makes sense after volume proves the need — and after operator #2 is in place.
Decide later — not urgent
AI automation layer specifics
Map it now, wire it in at week 3+. Do not automate a system that does not exist yet.
